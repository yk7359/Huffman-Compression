It is possible to compress a Huffman compress file that has already been compressed. In our experiment, we compressed the Calgary files a total of three times. Seen in the results, while the second compression lead to approximately 1.938% more compression, the third compression actually increased the size of the compressed file, with a compression average of -0.817%. This pattern is also seen in the Tree format, of which the second compression decreased the size while the third compression actually increased the size. This is likely due to the fact when it comes to the second compression, we are no longer compressing the original data, but rather the encoded data that resulted from the first Huffman compression, and so, while the first compression of the encoded data may lead to a tiny bit more compression, compressing the code of the encoded data, that is the third compression, will result in a bigger size due to the difference in the encoding. In other words, by adding more layers of encoding, we may end up making the file larger.  

The reason for not having efficient compression on compressed file is because, after we compressed the original file, we compress the file.hf again. Then the data in the file.hf.hf will pretty much be a random content in the file. Therefore, the frequencies for each ASCII value became more randomized. The point of Huffman coding is that we take the highest frequencies to get lowest length code. But if everything is being averaged, the compression will be inefficient. So, that is why if we compress again and again the compression became inefficient.
